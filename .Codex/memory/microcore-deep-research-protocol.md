# BuildWorld Micro-Core and Deep Research Protocol

Purpose: keep BuildWorld shipping from a tiny correct kernel, not from inflated scope or stale assumptions.

Micro-core rules:
1. One cycle, one core thesis.
2. Start with the smallest correct kernel before adding polish, integrations, or launch copy.
3. Separate the kernel from packaging. If the kernel is weak, do not hide it behind marketing.
4. Every promoted product claim must answer: what pain it solves, for whom, what proof exists, and why this build deserves to exist.
5. If a product is too crowded, too weak, or not buildable now, say so and do not force a ship cycle.

Deep research rules:
1. Before building or shipping, use at least 3 current sources when possible.
2. Include at least 1 official source such as framework docs, platform docs, or first-party product documentation.
3. Record source date, source type, competitor notes, distribution notes, contradiction notes, and confidence level.
4. Check current alternatives, pricing, launch channels, and platform constraints before claiming novelty or viability.
5. Treat BibleWorld and GhanaWorld inputs as hypotheses until benchmarked against real code and real market evidence.

Required cycle report blocks:
- Core Thesis
- Research Ledger
- Benchmark Check
- Reproducibility Block

Research Ledger minimum fields:
- Product concept
- Sources used
- Freshest source date
- Competitors checked
- Docs checked
- Distribution or pricing evidence
- Contradictions found
- Confidence level

Reproducibility Block minimum fields:
- Cycle ID
- Prompt version
- Freshest source date
- Benchmark items run
- Files updated
- Tests run
